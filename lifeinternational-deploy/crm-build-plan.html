Overview & North Star

Foundation ▼

The Challenge

  • 16 staff stewarding 50,000+ leaders
  • 170 countries, 10 global hubs
  • Legacy eTapestry not built for this
  • Data siloed across SharePoint docs
  • No live KPI visibility for donors

The Goal

  • Native Postgres CRM built for LI's reality
  • Automate digital relationship management
  • Track trainer lineage & multiplication chain
  • Surface live donor-facing impact metrics
  • Empower 16 staff to manage at global scale
🎯

North Star: "Because of your gift, 12 leaders were certified in Kenya this spring." — Every schema decision, every field, every dashboard should be evaluated against whether it gets Life International closer to being able to send that message to a donor automatically.

💡

Owen's framing: Hub structure is the multiplication strategy. Trainings are the bread and butter. Groups are the primary end-user value-add. The CRM underpins all three — most leaders won't even know they exist in it, but every meaningful interaction flows through it.

Data Architecture — Three Core + Three Supporting Record Types

Schema ▼
Record TypeWhat It RepresentsKey RelationshipsNotes
Constituent Core Individual people — leaders, trainers, partners Belongs to Hub (via country), linked to Organizations, linked to certifications, linked to events Primary record type. Most fields here.
Organization Core Partner churches and life-giving ministries Belongs to Hub (via country), linked to Constituents as members/leaders First-class entity, not just a field on Constituent
Hub Core 10 global hubs — lookup table driving auto-assignment Has many Constituents, has many Organizations — data derived not stored Must be its own table, not just a field value

Group Supporting Community discussion spaces Associated with Hub or global, auto-assigned from data or events Permission tiers for creation
⚠️

Critical schema risk: country_code must be a required field at ingestion — not optional. Geographic reporting (hub assignment, donor drill-down by country) cannot be rebuilt retroactively if this is missing from imported records.

User Roles & Permissions

Needs Discovery ▼
Part 1 — Existing Role Schema (from codebase)
RoleCodePriorityCurrent Access
Super Admin super_admin 3 Full system access, all users, all platform settings
Admin admin 2 Platform-wide user and configuration management
Editor editor 1 Content management — books, training modules, curriculum
Reader reader 0 Default role. Consumes content, joins groups, participates in training
Group Admin group_admin Contextual Scoped to a specific group via group_admins table. Manages members, invites, group events without platform-wide rights
💡

Trainer classification (Developing, Certified, Master Trainer) is intentionally NOT an AppRole — it is tracked as data on the certification record, not as a permission level. This is the correct architectural decision.

Part 2 — Proposed New Roles for CRM Layer
Proposed RoleCodeAccess LevelNotes
Hub Manager hub_manager Scoped read access to CRM data for their assigned hub only — constituent records, certification counts, training activity, hub dashboard. Cannot edit platform content or manage global users. Mirrors the group_admin pattern — implemented via a hub_managers join table (user_id + hub_id). A user can be a Reader globally but Hub Manager for their region.
CRM Admin crm_admin Full CRM read/write across all hubs — constituent records, certifications, organizations, reports, donor KPI layer. Does not include platform settings or user role management. For Zack, Owen, and key staff. Sits at priority level 2 alongside Admin but scoped to CRM rather than platform.
💡

The hub_manager role follows the same pattern as the existing group_admin — a join table scoping elevated rights to a specific entity. The hub_managers table needs: user_id, hub_id, assigned_at. One user can have multiple hub assignments if a Hub Chair coordinates across regions.

Part 3 — Row-Level Security (RLS) Requirement
⚠️

BLOCKING SCHEMA ITEM: Every CRM table requires Postgres row-level security (RLS) policies before any data is exposed. super_admin and admin see all rows. crm_admin sees all CRM rows. hub_manager sees only rows where country_code matches their assigned hub's countries. reader sees only their own record. Without RLS policies in place, hub filtering is a UI trick only — not actual data security. RLS must be designed in Phase 1 alongside the schema, not added later.

Part 4 — Open Discovery Questions

Certifications & Trainer Classification

Phase 1 — Core KPI ▼
🔑

Key rule: A person holds ONE certification level per curriculum at a time. But they can hold DIFFERENT levels across DIFFERENT curriculums simultaneously — e.g., Master Trainer in JOL and Developing Trainer in ELT. Schema requires a composite key: person_id + curriculum_id.

Curriculum Abbrev. Primary Purpose Developing Trainer Certified Trainer Master Trainer
Journey of a Life-Giver JOL Core entry-point curriculum. Bread and butter. Delivered in 3.5–4 days all at once. ✓✓✓
Equipping Life-Giving Trainers ELT Train-the-trainer curriculum. Equips people to deliver JOL. Fewer events, staff-led. ✓✓✓
Redemptive Conversations RC Specialized/elective. Not sequential with JOL. ✓✓✓
Post-Abortion Healing PAH Specialized/elective. Not sequential with JOL. ✓✓✓
🔗

The "Trained by" field: Every time a new certification is entered, it must link to the constituent record of the trainer who delivered it. This single field is what makes the multiplication chain traceable — who trained whom, going all the way up the lineage. It is the schema answer to Zack's ripple-effect tracking problem.
